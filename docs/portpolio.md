## 1. 모요 (Moyo)

### 한 줄 소개

Slack·밴드 UX를 분석하고 개선한 커뮤니티 웹 서비스 — 게시물·실시간 채팅·캘린더 기능을 통합하고 배포 자동화까지 구성

### 개요

| 항목      | 내용                                 |
| --------- | ------------------------------------ |
| 기간      | 2025.09.10 ~ 2025.12.06              |
| 팀 구성   | 3명 (개발 경험 전무한 학생들로 구성) |
| 역할      | 풀스택 (기여도 60~80%)               |
| 시연 영상 | https://youtu.be/EiS4uzRdQ9s         |

### 기술 스택

- **Frontend**: React, TypeScript, Redux Toolkit, TanStack Query, TailwindCSS, Shadcn/UI
- **Backend**: FastAPI, WebSocket, JWT, PostgreSQL
- **Infra**: AWS EC2, Docker Compose, Nginx, Cloudflare, GitHub Actions (CI/CD)

### 기술 아키텍처

```
React(TS, RTK, TanStack Query) ↔ FastAPI(JWT, WebSocket) ↔ PostgreSQL
AWS EC2(Docker Compose, Nginx) + Cloudflare + CI/CD
```

### 주요 기여

- **Frontend**: React+TypeScript 기반 전체 UI/UX 구현 (테마 포함), Redux Toolkit 전역 상태 관리, TanStack Query 서버 캐시/데이터 패칭 최적화
- **Backend**: FastAPI 기반 REST API 설계·구현 (계정탈퇴, 그룹참가, 게시물 CRUD, 친구추가 및 알림)
- **실시간**: FastAPI WebSocket 기반 실시간 채팅 구현
- **운영**: AWS EC2 + Docker Compose + Nginx 배포, 무료 도메인 + Cloudflare 구성
- **자동화**: 프론트/백 각각 CI/CD 파이프라인 구축 (GitHub Actions)

### 어려웠던 점

개발 경험이 전무한 팀원들과 함께 진행하면서, 중간중간 팀원들의 열정이 끊어지는 순간들이 있었다. 다른 팀원들이 다른 활동에 집중할 때도 일일 스크럼을 주도하고, 개발을 꾸준히 이어가며, 아이디어를 제시하면서 프로젝트가 멈추지 않도록 이끌어가는 것이 가장 힘든 부분이었다.

### 어떻게 해결했나

- 일일 스크럼 + 스프린트 회고를 지속적으로 주도
- 커밋 메시지 규칙과 작업 단위 공유(진행상황/이슈/결정사항)로 팀 흐름 유지
- 불편한 이슈도 명확히 정리·공유·합의하는 방식으로 소통

### 결과

- **완성된 핵심 기능**: 실시간 채팅 (WebSocket), 게시물 CRUD, 로그인·인증, 캘린더, 테마 변경, 게시물 작성 UI
- AWS EC2 + Docker Compose + Nginx + Cloudflare 배포 완료
- CI/CD 파이프라인 구축으로 프론트·백 자동 배포 환경 완성
- 일일 스크럼 기록과 커밋 히스토리로 팀 개발 흐름 문서화
- 개발이 재미있어진 계기가 된 프로젝트

### 배운 점

- 애자일 협업 프로세스 체득 (일일 스크럼·스프린트 계획)
- AI 활용 방식 전환: "안 쓰기" → "적극 활용 + 리팩토링으로 코드 내재화"
- 풀스택 역량 확장: 프론트 → 백엔드 → 배포/운영까지

### 추후 보완 과제

- 게시물 기능 고도화 (좋아요, 댓글, 동영상 첨부, 투표)
- 캘린더 UI/UX 재설계
- 더보기 탭 미구현 기능 완성

| 팀 구성    | 3명                                                                         |
| 역할       | 팀장·총괄 기획·풀스택 (기획~개발 전 과정 주도, 팀 구성·예산·교수 컨펌 포함) |

## 3. 코드바이트 (CodeByte)

### 한 줄 소개

개발자 면접 준비를 위한 CS 퀴즈 앱 — 4지선다·OX·단답형·매칭형 문제 유형, 스트릭 보상, 소셜 랭킹 기능 포함

### 개요

| 항목    | 내용                                   |
| ------- | -------------------------------------- |
| 기간    | 2026.03 ~ 2026.06 (1학기 팀 프로젝트)  |
| 팀 구성 | 4인 (신인규·임동환·정수한·황호태)      |
| 역할    | 프론트엔드 (UI/UX 구현·퀴즈 핵심 로직) |

### 기술 스택

- **Frontend**: React Native (Expo 54), TypeScript, Zustand, TanStack Query, Reanimated, Lottie
- **Backend (1차)**: Java 21, Spring Boot, Spring Security + JWT, Spring Data JPA, PostgreSQL, Docker
- **Backend (2차, 개인 리팩토링)**: Node.js, MongoDB, Vercel (서버리스 배포)

### 주요 기여

- 온보딩·홈·퀴즈·랭킹·상점·프로필 화면 UI/UX 전체 구현
- 커스텀 탭바 및 캐릭터·애니메이션 제작 (Reanimated SVG + Lottie)
- 퀴즈 핵심 로직 구현 (문제 생성, 채점, 결과 화면)
- 스트릭·보상 UI 구현 (streak screen, quest reward screen)
- 화면 성능 최적화 및 프론트엔드 리팩토링
- 프로젝트 종료 후 개인적으로 Node.js + MongoDB 서버리스 구조로 전환 및 Vercel 배포

### 기술적으로 인상 깊었던 부분

**퀴즈 로직 — 동일 난수(LCG) 기반 독립 생성**
팀원 중 알고리즘에 강한 백엔드 개발자가 설계한 구조로, 프론트와 백엔드가 동일한 시드를 공유해 통신 지연 없이 양측에서 똑같은 퀴즈를 독립적으로 생성. 프론트가 즉각 문제를 제공하고, 백엔드가 결과 제출 시 동일 로직으로 재현해 검증하는 구조.

**퀴즈 DB 구조**

- 과목 → 대주제 → 소주제(Concept)로 이어지는 계층형 트리 구조
- 리프 노드(자식 없는 최하위 개념)만 출제 타겟으로 한정해 일관성 확보
- Concept_Detail을 Key-Value 형태로 연결해 객관식·단답형·OX 다형적 문제 생성 지원

**Spring → Node.js + MongoDB 전환 이유**

- Isomorphic JS: FE(RN)와 BE(Node.js)가 같은 언어를 써서 난수 기반 알고리즘을 공통 모듈로 공유 가능 → 구현 불일치 리스크 원천 차단
- Schema-less: Key-Value 형태의 Concept_Detail을 MongoDB 도큐먼트로 직관적 저장 (RDB의 복잡한 EAV 패턴 불필요)
- 계층형 데이터 조회 최적화: 트리 구조를 Embedding·Aggregation으로 처리 → 복잡한 JOIN 없이 읽기 성능 극대화
- I/O 성능: Node.js 비동기 I/O 특성이 퀴즈 요청처럼 I/O 집중적인 워크로드에 적합

### 어려웠던 점

- 기획 문서(PRD·ERD·네이밍 컨벤션) 부재로 팀 협업 혼선 발생
- 백엔드 3명과 각각 다른 협업 방식(알고리즘 공동 설계 / API 문서 기반 소통 / AI 워크플로 문서 공유)으로 맞춰가야 했음

### 어떻게 해결했나

- 일일 스크럼과 스프린트 회고를 직접 주도해 팀 개발 흐름과 진행 상황을 지속적으로 정렬
- 이후 내가 기획 문서와 PRD를 직접 작성해 백엔드 팀원이 개발에만 집중할 수 있는 환경 구성
- AI로 작업 내용을 .md로 요약·저장해 다음 작업 세션에 컨텍스트로 활용하는 업무 방식 도입
- 업무를 작은 단위로 쪼개 수시로 공유하는 습관 형성

### 결과

- APK 배포 완료 (디자인 저작권 이슈로 스토어 출시 보류)
- Vercel 기반 서버리스 백엔드 개인 재구현 완료

### 배운 점

- **문서화의 힘**: 명확한 기획 문서가 있으면 유능한 개발자가 온전히 개발에 집중할 수 있다는 것을 직접 경험
- **백엔드 개발자의 사고방식 체득**: Java GC 원리, Spring vs Node.js vs MongoDB 차이를 잘하는 백엔드 개발자와의 대화로 배우며, 코드 하나하나의 실행 방식을 뜯어보는 사고 습관을 이해하게 됨
- **Git 브랜치 전략**: 팀 단위 브랜치 관리 경험
- **AI 활용 업무 방식**: 작업 내용을 AI로 요약해 .md로 저장 → 다음 세션 컨텍스트로 활용
- **업무 분배와 작은 단위 공유 습관**: 팀장으로서 업무를 배분하고, 진행 상황을 수시로 공유하는 협업 리듬 체득

## 4. 포폴 (Pofol)

### 한 줄 소개

개발자를 위한 포트폴리오 게시 + 팀원 모집 커뮤니티 플랫폼 — GitHub 연동, AI 요약·썸네일 자동 생성, 학교 이메일 OTP 인증 포함

### 개요

| 항목    | 내용                                                                       |
| ------- | -------------------------------------------------------------------------- |
| 기간    | 2026.03 ~ 2026.06 (캡스톤 디자인)                                          |
| 팀 구성 | 3인 (주민규·황호태·채수림)                                                 |
| 역할    | 풀스택 (백엔드 + 프론트엔드)                                               |
| 규모    | Java 파일 108개, DB 테이블 23개, API 엔드포인트 40개+, Next.js 페이지 17개 |

### 기술 스택

- **Frontend**: Next.js 16 (App Router), TypeScript 5, Tailwind CSS 4, Zustand, TanStack Query, Axios, Vercel Blob
- **Backend**: Java 21, Spring Boot 4.0, Spring Data JPA, Spring Security + JWT (jjwt 0.12), PostgreSQL, Redis (Lettuce), SpringDoc OpenAPI, Gradle
- **배포**: Vercel (FE) · Render + Docker Compose (BE + DB + Redis) → Superbase + Upstash(Redis) + Namecheap(DNS) 전환 시도
- **외부 연동**: GitHub OAuth, GitHub API, OpenAI DALL-E, Resend (이메일 발송), Vercel Blob (이미지 스토리지)

### 담당 역할 — 백엔드

- Post · Comment · 소셜 기능 (좋아요·북마크·팔로우) API 구현
- 학교 이메일 OTP 인증 시스템 구축 (Resend 이메일 발송)
- Spring Security + JWT 기반 인증 아키텍처 설계
- Redis를 활용한 OTP 및 세션 캐싱 처리
- ERD 설계부터 Entity·Repository·Service·Controller 전 과정 구현

### 담당 역할 — 프론트엔드

- 게시글 작성·수정·삭제 UI 전체 구현
- 팀 모집 지원하기 UI, OTP 인증 UI 구현
- GitHub OAuth 연동 UI 및 AI 기능(요약·썸네일 생성) 연동
- Vercel 배포 및 환경 설정

### 주요 기능

- **DISPLAY**: 포트폴리오 게시 및 피드 조회
- **RECRUIT**: 팀원 모집 게시글 및 지원 시스템
- **소셜 피드백**: 좋아요·북마크·팔로우·댓글
- **GitHub 연동**: OAuth 로그인 + GitHub API로 본인 레포지토리 불러오기 + README 기반 프로젝트 자동 파싱
- **AI 자동화**: OpenAI로 프로젝트 요약 + DALL-E 썸네일 자동 생성
- **재학생 인증**: 학교 이메일 OTP 인증

### 어려웠던 점

- DB 테이블 23개, API 40개+ 규모의 ERD 설계와 Spring 아키텍처 전 과정을 처음으로 직접 구현
- 무료 서버 환경(Render·Superbase·Upstash·Namecheap)의 한계를 직접 경험하며 Oracle·AWS가 필요한 이유를 몸으로 이해
- 다양한 외부 서비스(GitHub API, OpenAI, Resend, Vercel Blob) 동시 연동 및 오류 처리

### 어떻게 해결했나

- SpringDoc OpenAPI로 API 문서 자동화해 팀 간 소통 효율 확보
- Next.js Middleware·Proxy로 API 요청을 서버 사이드에서 처리해 인증 토큰 관리 일원화
- 무료 배포 환경 한계 부딪히며 유료 인프라의 필요성과 차이를 실질적으로 학습

### 결과

- 캡스톤 디자인 프로젝트 완성
- GitHub README 기반 AI 자동 요약·썸네일 생성 기능 구현 완료

### 배운 점

- **Spring 풀사이클 체득**: ERD 설계 → Entity → Repository → Service → Controller 전 과정을 직접 구현하며 백엔드 아키텍처 구조 이해
- **외부 서비스 통합 경험**: GitHub API, OpenAI, Resend, Vercel Blob 등 다양한 서비스를 실제로 연동하며 통합 개발 감각 습득
- **무료 vs 유료 인프라 차이 체감**: Render·Superbase·Upstash·Namecheap으로 배포하며 Oracle·AWS의 안정성과 필요성을 직접 이해
- **Next.js Middleware·Proxy 활용**: 서버 사이드 API 프록시 처리 및 인증 토큰 관리 흐름 설계

## 5. 강의실 자동점검 시스템

### 한 줄 소개

교학팀 근로학생이 매일 20분~1시간씩 수기로 짜던 강의실 점검 배정 업무를 자동화한 React SPA — 시간표를 붙여넣으면 4가지 규칙을 만족하는 배정표가 즉시 생성

### 개요

| 항목    | 내용                                                            |
| ------- | --------------------------------------------------------------- |
| 기간    | 2026.03 (초기 개발 4일) → 2026.05 현재까지 운영·유지보수 중     |
| 팀 구성 | 개인 프로젝트                                                   |
| 역할    | 기획·설계·개발 전담                                             |
| 배경    | 교학1팀 근로 중 수작업 업무 직접 발견 → 자동화 결정             |
| 현황    | 실제 근로학생들이 사용 중, 방학 중 협업을 통해 확장성 개선 예정 |

### 기술 스택

- **Frontend**: React 18, Vite 6, 순수 CSS (외부 UI 라이브러리 없음)
- **구조**: App.jsx (전역 상태 관리), utils.js (파싱·시간 연산·자동 배정 알고리즘), schedule.js (요일별 근무자 시간표·색상 매핑)
- **배포**: (추가 예정)

### 자동 배정 알고리즘 — 4가지 핵심 규칙

| 규칙                       | 코드 구현                                                      |
| -------------------------- | -------------------------------------------------------------- |
| 균등 배분 (±1)             | maxLoad = ceil(평균)+1 상한 + imbalance 점수화                 |
| 한 번에 점검 (이동 최소화) | commonWindows 교집합 추적 + tripPenalty 10000                  |
| 층수 최소화 (≤2개 층)      | maxFloors = 2 상한 + 인접 층 거리 기반 floorPenalty            |
| 퇴근 전 여유 (50분)        | EARLY_FINISH_MIN으로 근무 종료시각 앞당겨 effective range 계산 |

### 알고리즘 동작 단계

1. **제약 전파 정렬** — 배정 가능한 근무자 수가 적은(까다로운) 강의실을 먼저 배정
2. **그리디 배정 (greedyRun)** — 근무자별 공통 시간 교집합(commonWindows)을 유지하며 실시간 이동 횟수 추정. 층수 패널티는 단순 초과가 아닌 거리 기반 차등 (인접 층 500점, 멀수록 최대 9000점)
3. **로컬 서치 최적화 (localSearch)** — 그리디 결과를 스왑·이동으로 미세 조정. 6개 지표를 사전식(lexicographic) 비교: [1회이동 위반자 → 배정수 초과자 → 층수 초과자 → 불균형 → 층범위 합산 → 총이동횟수]
4. **멀티 리스타트 (20회)** — 정렬 순서를 셔플해 다른 출발점으로 20번 시도, 최적 결과 채택. 4개 핵심 지표가 모두 0이면 조기 종료

### 설계에서 인상 깊은 포인트

- **단일 숫자 점수 대신 벡터 점수 + 사전식 비교**: 규칙 간 우선순위(이동 최소화 > 균등 배분 > 층 인접성)를 코드로 명확하게 표현
- **공통 시간 교집합 추적**: 이동 횟수를 배정 시점에 실시간 추정하고, countTrips로 사후 검증도 별도 수행해 그리디 근사 오차 보정
- **행사(이벤트) 반영**: computeEffectiveSlots가 강의실 시간 슬롯에서 행사 시간을 실시간으로 빼고, 행사 추가 시 기존 배정도 자동 재검토

### 어려웠던 점

- 실무 담당자의 암묵적인 규칙(수기로 짜던 기준)을 코드로 명확히 정의하는 과정
- 단순 그리디로는 최적해를 보장할 수 없어 멀티 리스타트 + 로컬 서치 구조 설계 필요

### 결과

- 매일 20분~1시간 걸리던 수작업 배정 업무 즉시 자동화
- 실제 근로학생들이 사용하며 발견된 오류 지속 수정 중
- 행사 일정 수동 입력 → 복사·붙여넣기 자동화로 개선
- 방학 중 협업을 통해 학기별·근로학생 교체 대응 가능한 구조로 확장 예정

### 배운 점

- 실무에서 발생하는 문제를 직접 발견하고 정의하는 경험
- 알고리즘 설계 시 단순 점수화가 아닌 우선순위 기반 벡터 비교의 효과
- 사용자(실무자) 니즈를 파악하고 규칙으로 변환하는 과정

## 6. 영화 일기 (Yeflix)

### 한 줄 소개

영화를 봤지만 기억 못해 대화에서 애먹던 경험에서 출발 — 영화 검색·별점·감상문을 기록하고 OTT 바로가기까지 제공하는 개인 영화 다이어리 서비스

### 개요

| 항목    | 내용                                                                 |
| ------- | -------------------------------------------------------------------- |
| 기간    | 2026.04 (초기 개발 3일, 이후 점진적 업데이트 중)                     |
| 팀 구성 | 개인 프로젝트                                                        |
| 역할    | 기획·개발·배포 전담                                                  |
| 배경    | 영화를 좋아하는 친구들과 더 깊은 대화를 나누고 싶어서 직접 제작      |
| 현황    | 개인적으로 지속 사용 중, 모바일 반응형·찜 기능 등 점진적 업데이트 중 |

### 기술 스택

- **Framework**: Next.js 14 (App Router), JavaScript (JSX)
- **스타일링**: Tailwind CSS
- **인증/DB**: Firebase Authentication (Google 로그인) + Firestore
- **외부 API**: TMDB (The Movie Database)
- **배포**: Vercel

### 주요 기능

- 현재상영·인기·예정 영화 목록 조회 및 검색
- 영화 상세 페이지 (출연진·OTT 바로가기·별점)
- 영화 일기 작성 (별점·날짜·감상문·태그)
- 찜하기(Watchlist) 기능
- Google 로그인 기반 개인 데이터 관리

### 설계에서 인상 깊은 포인트

- **Firestore 합성 키 전략**: 문서 ID를 `${userId}_${movieId}`로 구성해 "영화당 일기·찜 1개" 제약을 스키마 레벨에서 자연스럽게 구현 (upsert 패턴)
- **복합 인덱스 회피**: 정렬을 클라이언트 사이드로 위임해 Firestore 인덱스 비용 절감 — 소규모 개인 프로젝트에 적합한 트레이드오프 의식적으로 선택
- **단일 fetcher 패턴**: TMDB API 요청을 하나의 fetcher 함수로 감싸 api_key·language 파라미터를 공통 주입

### 배운 점

- Firebase Auth + Firestore를 활용한 인증·데이터 관리 전반 경험
- 외부 API(TMDB) 연동 및 래퍼 함수 설계
- 트레이드오프를 의식적으로 선택하는 설계 감각 (인덱스 비용 vs 클라이언트 정렬)
- 내가 직접 쓰는 서비스를 만들며 사용자 관점에서 지속적으로 개선하는 경험
